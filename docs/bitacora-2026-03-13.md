## 4. Problema con credenciales PostgreSQL y resolucion

Problema encontrado:

- la aplicacion no podia iniciar porque fallaba la autenticacion PostgreSQL para el usuario `postgres`

Lo que se verifico realmente:

- PostgreSQL 16 estaba instalado y levantado localmente
- `psql.exe` existia en la instalacion local
- el servicio Windows `postgresql-x64-16` estaba corriendo
- la conexion al servidor llegaba correctamente
- fallaba especificamente el password para `postgres`

Decision tomada:

- no reutilizar `postgres`
- no adivinar credenciales
- no fingir la creacion de DB/usuario
- preparar el backend para usar un usuario dedicado y documentar el setup manual

Resultado:

- se dejo el backend apuntando a `sentimospadel` en `public`
- se documento el enfoque con usuario dedicado local
- el proyecto quedo desacoplado del uso directo del superusuario `postgres`

## 5. Problema con Flyway en Spring Boot 4

Problema encontrado:

- la app conectaba a PostgreSQL
- el schema `public` estaba vacio
- Hibernate fallaba con:
  - `Schema validation: missing table [clubs]`

Analisis realizado:

- se inspecciono `pom.xml`
- se reviso `application.yml`
- se verifico la ruta de migraciones
- se verifico que el archivo existia en:
  - `src/main/resources/db/migration`
- se verifico que el recurso quedaba empaquetado en:
  - `target/classes/db/migration`
- se revisaron exclusiones de auto-configuracion
- se confirmo que las exclusiones en tests no afectaban runtime

Archivo verificado:

- [V1__init_core_schema.sql](/C:/Users/natal/sentimospadel/backend/src/main/resources/db/migration/V1__init_core_schema.sql)

Root cause:

- el proyecto usaba Spring Boot `4.0.3`
- en Boot 4 el soporte de Flyway no quedaba cubierto solo con `flyway-core`
- faltaba el starter/modulo correcto de Spring Boot para Flyway

Resolucion:

- se reemplazo el uso directo de `flyway-core`
- se agrego `spring-boot-starter-flyway`
- se mantuvo `flyway-database-postgresql`

Archivo afectado:

- [pom.xml](/C:/Users/natal/sentimospadel/backend/pom.xml)

Resultado posterior:

- Flyway empezo a correr antes que Hibernate
- se creo `flyway_schema_history`
- se aplico `V1__init_core_schema.sql`
- Hibernate pudo validar el esquema correctamente
